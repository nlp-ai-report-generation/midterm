## 1. 꼭 필요한 핵심 기능

### 1-1. STT 스크립트 전처리 파이프라인

원본 STT 데이터는 오탈자, 띄어쓰기 오류, 화자 구분 누락 등이 많다. KoNLPy(Okt 또는 Mecab)를 활용한 형태소 분석 기반 정제가 선행되어야 하며, 타임스탬프가 포함된 경우 이를 파싱하여 발화 속도(분당 발화량) 계산에 활용해야 한다. 화자 분리(강사 vs 수강생)가 되어 있지 않다면, 발화 패턴이나 별도 diarization 처리를 고려해야 한다.

### 1-2. 반복 표현 검출 (카테고리 1: 언어 표현 품질)

"이제", "그래서", "자" 같은 습관성 filler word의 빈도를 n-gram 분석으로 카운팅하고, 동일 문장이나 구절의 반복도 탐지해야 한다. 단순 빈도뿐 아니라 시간 구간별 밀도를 측정하면 "어느 시점에서 반복이 심해지는지"까지 파악할 수 있어 훨씬 유용하다.

### 1-3. 강의 구조 분석 (카테고리 2: 강의 도입 및 구조)

강의 시작부에 학습 목표 안내가 있는지, 전날 복습 연계가 있는지, 마무리 요약이 있는지를 LLM 프롬프트로 판별한다. 스크립트를 도입/본문/마무리 세그먼트로 분할한 뒤 각 구간에 대해 구조적 요소 존재 여부를 체크하는 방식이 효과적이다.

### 1-4. 개념 설명 명확성 평가 (카테고리 3)

핵심 용어가 처음 등장할 때 정의가 수반되는지, 비유나 실생활 예시가 활용되는지를 LLM이 판단한다. 이 부분은 규칙 기반으로는 한계가 있어 LLM 기반 평가가 필수적이며, 프롬프트에 체크리스트의 세부 기준을 명시적으로 포함시켜야 정확도가 올라간다.

### 1-5. 예시·실습 연계 및 상호작용 분석 (카테고리 4, 5)

"한번 해보세요", "되셨어요?", "이해하셨나요?" 같은 참여 유도 및 이해 확인 패턴을 검출하고, 이론 설명 후 실습으로의 전환이 자연스럽게 이루어지는지를 분석한다. 수강생 질문에 대한 응답 충분성은 화자 분리가 전제되어야 분석 가능하다.

### 1-6. 점수 산출 및 리포트 생성

각 평가 항목별 1~5점 점수를 산출하되, 체크리스트에 명시된 가중치(높음/중간/낮음)를 반영한 가중 평균을 계산해야 한다. 최종 리포트에는 총점, 항목별 점수, 잘한 점, 개선 필요 사항, 추천 개선 방향이 포함되어야 하며, python-docx 또는 ReportLab로 PDF/DOCX 형태로 출력한다.

## 2. 기술 스택 조합 시 주의할 점

### 2-1. LangChain vs LlamaIndex 선택 기준

둘 다 쓸 필요는 없다. 이 프로젝트는 긴 강의 스크립트를 청크 단위로 분석한 뒤 결과를 종합하는 패턴이므로, 프롬프트 체이닝과 순차 분석에 강한 **LangChain이 더 적합**하다. LlamaIndex는 대량 문서에서 검색(RAG)이 핵심일 때 유리한데, 이 프로젝트는 검색보다 분석이 중심이라 오버엔지니어링이 될 수 있다.

### 2-2. LLM API 비용 관리

강의 스크립트 하나가 수만 토큰에 달할 수 있다. 전체 스크립트를 한 번에 LLM에 넘기면 비용이 급증하므로, **규칙 기반으로 처리 가능한 항목(반복 표현 빈도, 발화 속도 등)은 Python/KoNLPy로 먼저 처리**하고, LLM은 판단이 필요한 항목(구조 분석, 명확성 평가 등)에만 사용하는 하이브리드 접근이 중요하다. 또한 스크립트를 시간 구간별로 청크를 나눠서 분석한 뒤 최종 종합하는 **Map-Reduce 패턴**을 적용할 것을 권장한다.

### 2-3. KoNLPy 환경 이슈

KoNLPy는 Java(JDK) 의존성이 있어 팀원 전체의 개발 환경 세팅이 까다롭다. 특히 Mecab을 쓸 경우 OS별 설치 방법이 다르므로, **Docker로 개발 환경을 통일**하는 것을 강력히 권장한다. 혹은 Okt(순수 Java 기반)를 사용하면 설치 문제가 줄어든다.

### 2-4. Streamlit의 한계 인지

Streamlit은 빠른 프로토타이핑에 좋지만, 다중 페이지 구성이나 복잡한 상태 관리에는 한계가 있다. 리포트 대시보드 수준이면 충분하지만, 만약 사용자 인증이나 DB 연동이 필요해지면 FastAPI + React 같은 구조로의 전환을 염두에 둘 것. Gradio는 ML 데모용으로는 좋으나 커스텀 UI에 제약이 많아, 이 프로젝트에서는 **Streamlit이 더 나은 선택**이다.

### 2-5. GPT-4o vs Claude API

둘 중 하나를 주 모델로 정하고, 다른 하나는 fallback으로 두는 것이 좋다. 프롬프트 형식이 미묘하게 다르므로(Claude는 XML 태그 기반 프롬프트에 강함, GPT-4o는 JSON 구조에 강함), **추상화 레이어를 두어 모델 교체가 용이하도록** 설계할 것.

## 3. 4인 역할 분배

### 업무 배분 개요

| 담당자 | 핵심 업무 | 난이도 | 코드 볼륨 | 총 업무량 |
|--------|-----------|--------|-----------|-----------|
| A | 전처리 파이프라인 + 인프라 | 높음 | 중간 | ★★★ |
| B | 규칙 기반 분석 + 점수 산출 | 중간 | 높음 | ★★★ |
| C | LLM 기반 분석 + 프롬프트 엔지니어링 | 높음 | 낮음 | ★★★ |
| D | 리포트 생성 + Streamlit UI | 중간 | 높음 | ★★★ |

### 담당자 A — 전처리 파이프라인 + 인프라

STT 원본 데이터를 분석 가능한 상태로 정제하는 전체 흐름을 담당한다. 텍스트 클리닝, 타임스탬프 파싱, 화자 분리가 핵심이고, 여기에 Docker 환경 구성과 프로젝트 초기 세팅(gitignore, 설정 관리 등)까지 맡는다. 전처리 품질이 이후 모든 분석의 정확도를 좌우하므로 난이도는 높지만 코드 볼륨은 상대적으로 적다. 인프라 세팅을 함께 맡겨서 총량을 맞춘다.

### 담당자 B — 규칙 기반 분석 + 점수 산출

Python과 KoNLPy만으로 처리 가능한 분석 항목들을 담당한다. 반복 표현 검출(n-gram 빈도 분석), 발화 속도 계산, 상호작용 패턴 검출("되셨어요?", "해보세요" 등), 그리고 전체 평가 항목의 가중 점수를 종합 산출하는 스코어링 로직까지 맡는다. LLM 없이 동작하는 모듈이라 독립적으로 개발·테스트할 수 있고, 항목 수가 많아서 볼륨이 충분하다.

### 담당자 C — LLM 기반 분석 + 프롬프트 엔지니어링

사람의 판단이 필요한 정성적 평가 항목을 LLM으로 처리한다. LLM API 추상화 레이어 구축, 프롬프트 템플릿 설계, LangChain 체인 구성이 핵심이고, 강의 구조 분석(도입·복습·요약 유무)과 개념 설명 명확성 평가(정의 제공 여부, 비유 활용 여부)를 구현한다. 프롬프트 튜닝에 반복 실험이 필요해서 시간이 많이 들지만 코드 볼륨 자체는 적으므로 균형이 맞는다.

### 담당자 D — 리포트 생성 + Streamlit UI

분석 결과를 사용자에게 전달하는 출력 레이어 전체를 담당한다. DOCX/PDF 리포트 내보내기와 Streamlit 기반 대시보드(업로드, 결과 시각화, 분석 이력) 구현이 핵심이다. UI 작업은 눈에 보이는 결과물이 많아 체감 볼륨이 크지만, 비즈니스 로직이 적고 다른 모듈의 출력을 조합하는 역할이라 난이도는 상대적으로 낮다.

## 5. 협업 가이드

### 5-1. 모듈 간 인터페이스를 먼저 합의할 것

4명이 독립적으로 개발하려면 각 모듈의 입출력 형식이 사전에 확정되어야 한다. `docs/interface_contract.md`에 아래와 같은 공통 규약을 프로젝트 초기에 함께 정의하고 시작하는 것을 권장한다.

```python
# 모든 분석 함수의 공통 반환 형식 예시
{
    "category": "언어 표현 품질",
    "item": "불필요한 반복 표현",
    "score": 3,
    "weight": "높음",
    "evidence": ["'이제' 47회 사용", "'그래서' 32회 사용"],
    "detail": "분당 평균 2.3회의 filler word가 감지되었습니다."
}
```

### 5-2. 브랜치 전략

4명이면 Git Flow의 `release` 브랜치까지는 필요 없다. `main` → `develop` → `feature/a-preprocessing`, `feature/b-rule-analysis` 패턴이면 충분하다. 각자의 feature 브랜치에서 작업하고 develop으로 PR을 올려서 최소 1명 리뷰 후 머지하는 규칙을 세울 것.

```
main ─────────────────────────────── 배포용 (안정 버전만)
  └── develop ────────────────────── 통합 개발 브랜치
        ├── feature/a-preprocessing
        ├── feature/b-rule-analysis
        ├── feature/c-llm-analysis
        └── feature/d-report-ui
```

### 5-3. 작업 순서 권장

A가 가장 먼저 시작해야 한다. 전처리 모듈의 출력이 B와 C의 입력이 되므로, 프로젝트 초기 1~2일은 A가 샘플 데이터 기반으로 전처리 파이프라인의 프로토타입과 출력 포맷을 확정하고, B·C는 그 사이에 `data/sample/`의 샘플 데이터로 각자 분석 로직을 설계하는 것이 효율적이다. D는 UI 레이아웃과 리포트 템플릿을 먼저 잡아두면 된다.

### 5-4. 보안 주의사항

`.env`에 API 키를 넣고 `.gitignore`에 반드시 추가할 것. 실수로 커밋되는 것을 방지하려면 `pre-commit` 훅으로 시크릿 검출 도구(예: `detect-secrets`)를 걸어두면 좋다.
