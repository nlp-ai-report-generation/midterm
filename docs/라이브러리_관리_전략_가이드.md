# 라이브러리 관리 전략 가이드

> **프로젝트:** AI 강의 분석 리포트 생성기  
> **전제:** OS별 환경 세팅은 완료된 상태 (Windows/macOS 가이드 참고)

---

## 1. 왜 관리가 필요한가

4명이 각자 다른 모듈을 개발하면 필요한 라이브러리가 다르다. 담당자 A는 KoNLPy와 pandas가 필요하고, 담당자 C는 LangChain과 OpenAI SDK가 필요하다. 전원이 모든 패키지를 설치하면 불필요한 의존성 충돌이 생길 수 있고, 반대로 각자 알아서 설치하면 버전이 제각각이 되어 통합 시 문제가 발생한다.

이를 해결하기 위해 **계층형 requirements 구조**를 사용한다.

---

## 2. requirements 파일 구조

```
init/
├── requirements/
│   ├── base.txt               # 전원 공통 (필수)
│   ├── preprocessing.txt      # 담당자 A
│   ├── rule_analysis.txt      # 담당자 B
│   ├── llm_analysis.txt       # 담당자 C
│   ├── report_ui.txt          # 담당자 D
│   └── dev.txt                # 개발 도구 (전원 공통)
└── requirements.txt           # 전체 통합 (배포/CI용)
```

각 모듈별 파일은 `-r base.txt`로 공통 패키지를 상속받는다. 따라서 자기 파일 하나만 설치하면 공통 패키지도 함께 설치된다.

## 4. 설치 방법

venv가 활성화된 상태에서 자기 담당 파일만 설치한다.

```bash
# 예: 담당자 B
pip install -r requirements/rule_analysis.txt
pip install -r requirements/dev.txt
```

통합 테스트나 배포 시에는 전체를 설치한다.

```bash
pip install -r requirements.txt
```

---

## 5. 새 라이브러리를 추가할 때

### 5-1. 설치 후 자기 requirements 파일에 추가

```bash
# 예: 담당자 B가 scikit-learn을 추가 설치
pip install scikit-learn
```

설치 후 `requirements/rule_analysis.txt`에 아래 한 줄을 추가한다.

```txt
scikit-learn>=1.4.0
```

### 5-2. pip freeze를 사용하지 않는 이유

`pip freeze > requirements.txt`를 실행하면 직접 설치한 패키지뿐 아니라 그 패키지의 의존성까지 전부 나열된다. 예를 들어 `langchain`만 설치해도 freeze 결과에 수십 개의 하위 패키지가 포함되어 파일이 비대해지고 관리가 어려워진다.

원칙은 **직접 설치한 패키지만 수동으로 추가**하는 것이다. 하위 의존성은 pip이 자동으로 해결해준다.

## 6. 다른 팀원의 변경사항 반영

### 6-1. develop 브랜치를 pull한 뒤 재설치

```bash
git checkout develop
git pull origin develop

# 변경된 requirements 반영
pip install -r requirements/base.txt
pip install -r requirements/본인모듈.txt
```

### 6-2. 통합 전에 전체 설치로 확인

PR 머지 전에 전체 패키지가 충돌 없이 설치되는지 확인한다.

```bash
pip install -r requirements.txt
```

---

## 7. 버전 충돌이 발생하면

두 모듈이 같은 패키지의 다른 버전을 요구하는 경우 (예: A는 `pandas>=2.0`, D는 `pandas<2.0`), **높은 버전으로 통일**하는 것을 원칙으로 한다.

해결이 안 되면 아래 순서로 대응한다.

1. `pip install 패키지명 --upgrade`로 최신 버전 설치 후 양쪽 코드에서 호환성을 확인한다.
2. 특정 버전 고정이 필요하면 `base.txt`에 명시하여 전원이 동일 버전을 사용하도록 한다.
3. 그래도 해결되지 않으면 팀 회의에서 논의한다.

## 8. 환경 검증 스크립트

프로젝트 루트에 아래 파일을 둔다. 팀원이 환경 세팅 후 바로 확인할 수 있다.

### scripts/check_env.py

```python
"""환경 세팅 검증 스크립트 — python scripts/check_env.py 로 실행"""
import sys
import shutil


def check_python():
    v = sys.version_info
    ok = v.major == 3 and v.minor >= 11
    status = "OK" if ok else "FAIL (3.11 이상 필요)"
    print(f"[Python]  {v.major}.{v.minor}.{v.micro}  ...  {status}")
    return ok


def check_java():
    java = shutil.which("java")
    if java:
        import subprocess
        result = subprocess.run(
            ["java", "-version"], capture_output=True, text=True
        )
        version_line = result.stderr.split("\n")[0]
        print(f"[Java]    {version_line}  ...  OK")
        return True
    else:
        print("[Java]    not found  ...  FAIL (JDK 11 이상 설치 필요)")
        return False


def check_venv():
    in_venv = hasattr(sys, "real_prefix") or (
        hasattr(sys, "base_prefix") and sys.base_prefix != sys.prefix
    )
    status = "OK" if in_venv else "FAIL (venv 활성화 필요)"
    print(f"[venv]    {'활성화됨' if in_venv else '비활성'}  ...  {status}")
    return in_venv


def check_packages():
    packages = {
        "dotenv": "python-dotenv",
        "yaml": "pyyaml",
        "konlpy": "konlpy",
        "pandas": "pandas",
        "langchain": "langchain",
        "streamlit": "streamlit",
        "docx": "python-docx",
    }
    results = {}
    for module, name in packages.items():
        try:
            __import__(module)
            results[name] = "OK"
        except ImportError:
            results[name] = "미설치 (담당 모듈이면 설치 필요)"

    print("\n[패키지 상태]")
    for name, status in results.items():
        print(f"  {name:20s} ... {status}")


if __name__ == "__main__":
    print("=" * 55)
    print("  환경 세팅 검증")
    print("=" * 55)
    check_python()
    check_java()
    check_venv()
    check_packages()
    print("=" * 55)
```

## 9. 요약

| 상황 | 행동 |
|------|------|
| 처음 환경 세팅 | OS별 가이드 → venv 생성 → 자기 requirements 설치 |
| 새 패키지 추가 | `pip install` → 자기 requirements 파일에 수동 추가 |
| 다른 팀원 PR 머지 후 | `git pull` → `pip install -r requirements/본인모듈.txt` |
| 통합 테스트 전 | `pip install -r requirements.txt`로 전체 설치 |
| 버전 충돌 | 높은 버전으로 통일, 안 되면 팀 회의 |
