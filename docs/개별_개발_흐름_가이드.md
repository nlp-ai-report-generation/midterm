# 개별 개발 흐름 가이드

> **프로젝트:** AI 강의 분석 리포트 생성기  
> **대상:** 팀원 전원 (A, B, C, D)  
> **핵심 원칙:** 평소 개발은 venv, 통합 확인은 Docker

---

## 1. venv와 Docker, 각각 언제 쓰는가

venv와 Docker는 둘 중 하나만 고르는 것이 아니라 **쓰는 시점이 다르다.** venv는 "내가 코드 짤 때" 쓰는 것이고, Docker는 "팀 전체가 합칠 때" 쓰는 것이다.

비유하자면 venv는 각자의 작업 책상이고, Docker는 팀 공용 작업실이다. 평소에는 각자 책상에서 작업하다가, "이거 합쳐서 돌려보자" 할 때 공용 작업실에 모이는 식이다.

| 기준 | venv | Docker |
|------|------|--------|
| 사용 빈도 | 매일, 하루 종일 | 가끔 (통합 테스트, 배포) |
| 용도 | 내 모듈 개발·디버깅·단위 테스트 | 전체 통합 테스트·환경 차이 해소 |
| 속도 | 빠름 (코드 고치고 바로 실행) | 느림 (컨테이너 기동 시간 있음) |
| 필수 여부 | 필수 | 권장 (없어도 개발은 가능) |

### venv를 쓰는 상황

일상적인 개발 작업 전부다. 코드 작성, 디버깅, 단위 테스트, 자기 모듈 실행 확인 등이 해당된다. VS Code 같은 IDE도 venv를 인식해서 자동완성이나 린팅이 잘 동작한다. 가볍고 빠르기 때문에 코드 한 줄 고치고 바로 실행해보는 개발 사이클에 적합하다.

### Docker를 쓰는 상황

사용 시점이 한정적이다. 모든 모듈을 합쳐서 통합 테스트를 돌릴 때, "내 컴에서는 되는데 네 컴에서는 안 돼"를 확인할 때, 그리고 최종 배포할 때 정도다.

### venv만으로는 부족한 이유

venv는 Python 패키지만 격리해준다. 하지만 이 프로젝트는 Java(KoNLPy용)가 필요하고, 팀원마다 OS가 다르다. venv로는 "담당자 A의 Windows에서는 KoNLPy가 되는데 담당자 C의 macOS에서는 Java 경로 문제로 안 된다" 같은 상황을 막을 수 없다. Docker는 이런 OS 레벨 차이까지 통일시켜준다.

단, 팀 사정상 Docker 도입이 부담스럽다면 venv만으로도 프로젝트 진행은 충분히 가능하다. Docker는 필수가 아니라 권장이다.

## 3. 아침 — 작업 시작 (venv)

터미널을 열고 아래 순서를 따른다.

### 3-1. 프로젝트 폴더로 이동

**macOS:**
```bash
cd ~/Documents/init
```

**Windows:**
```
cd C:\Users\본인계정\Documents\init
```

### 3-2. venv 활성화

**macOS:**
```bash
source .venv/bin/activate
```

**Windows (CMD):**
```
.venv\Scripts\activate.bat
```

**Windows (PowerShell):**
```
.venv\Scripts\Activate.ps1
```

터미널 앞에 `(.venv)`가 표시되면 정상이다.

### 3-3. 최신 코드 가져오기

```bash
git checkout develop
git pull origin develop
```

### 3-4. 자기 feature 브랜치로 이동

```bash
# 이미 있는 브랜치로 이동
git checkout feature/b-rule-analysis

# 새 작업을 시작할 때 (develop에서 분기)
git checkout develop
git checkout -b feature/b-새기능이름
```

### 3-5. requirements 변경 확인

다른 팀원이 requirements 파일을 수정했을 수 있다. pull 후에 재설치하면 안전하다.

```bash
pip install -r requirements/base.txt
pip install -r requirements/본인모듈.txt
```

> 매번 하기 번거롭다면, git pull 후 `requirements/` 폴더에 변경이 있는지 확인하고 변경이 있을 때만 실행해도 된다.
>
> ```bash
> git diff HEAD~1 --name-only | grep requirements
> ```
>
> 출력이 있으면 재설치가 필요한 것이다.

## 4. 낮 — 코드 작성과 테스트 반복 (venv)

이 단계가 하루 중 가장 오래 걸리는 핵심 작업 시간이다. Docker는 쓰지 않는다.

### 4-1. 코드 작성 → 바로 실행

```bash
# 예: 담당자 B가 반복 표현 검출 모듈 실행
python -m src.rule_analysis.repetition

# 예: 담당자 C가 LLM 분석 모듈 실행
python -m src.llm_analysis.structure_eval

# 예: 담당자 D가 Streamlit 앱 실행
streamlit run app/main.py
```

코드를 수정하면 저장 후 바로 다시 실행해서 결과를 확인한다. venv 환경에서는 이 사이클이 1~2초 안에 끝난다.

### 4-2. 단위 테스트 실행

```bash
# 내 모듈의 테스트만 실행
pytest tests/test_rule_analysis.py -v

# 특정 테스트 함수만 실행
pytest tests/test_rule_analysis.py::test_filler_word_count -v

# 전체 테스트 실행 (가끔)
pytest tests/ -v
```

### 4-3. 담당자별 개발 팁

**담당자 A (전처리 + 인프라)**  
`data/sample/`에 있는 샘플 STT 스크립트로 전처리 결과를 확인한다. 출력 포맷이 바뀌면 B, C에게 반드시 공유한다.

```bash
python -m src.preprocessing.text_cleaner
# 결과물을 data/processed/에 저장하고 확인
```

**담당자 B (규칙 기반 분석)**  
A의 전처리 출력이 아직 없다면 `data/sample/`의 원본 데이터로 먼저 개발한다. 나중에 A의 출력 포맷에 맞춰 입력 부분만 수정하면 된다.

```bash
python -m src.rule_analysis.repetition
python -m src.rule_analysis.speech_rate
```

**담당자 C (LLM 분석)**  
LLM API 호출은 비용이 발생하므로, 프롬프트 테스트 시 짧은 샘플 텍스트로 먼저 확인한다. `.env` 파일에 API 키가 설정되어 있어야 한다.

```bash
# .env 파일 확인
cat .env   # macOS
type .env  # Windows

# 짧은 텍스트로 프롬프트 테스트
python -m src.llm_analysis.structure_eval
```

**담당자 D (리포트 + UI)**  
B, C의 분석 결과가 아직 없다면 더미 데이터를 만들어서 UI를 먼저 개발한다. `docs/interface_contract.md`에 정의된 공통 반환 형식을 참고하여 더미 데이터를 구성한다.

```bash
# Streamlit 앱 실행
streamlit run app/main.py
# 브라우저에서 http://localhost:8501 확인
```

## 5. 저녁 — 커밋 및 통합 확인

### 5-1. 자기 모듈 최종 테스트 (venv)

커밋 전에 자기 모듈 테스트가 전부 통과하는지 확인한다.

```bash
pytest tests/test_본인모듈.py -v
```

### 5-2. Docker로 통합 테스트 (선택 — 권장)

PR을 올리기 전에, Docker 환경에서 전체 테스트를 한 번 돌려보면 환경 차이로 인한 문제를 미리 잡을 수 있다.

```bash
docker compose run --rm app pytest tests/ -v
```

Docker에서 통과하면 안심하고 PR을 올린다. Docker에서만 실패한다면 로컬 venv 환경에 특정 설정이 있어서 된 것이므로, 원인을 파악해야 한다.

> Docker가 아직 세팅되지 않았거나 부담스럽다면 이 단계는 건너뛰어도 된다. 대신 PR 리뷰어가 자기 환경에서 돌려보는 것으로 대체할 수 있다.

### 5-3. 커밋 및 푸시

```bash
# 변경 사항 확인
git status
git diff

# 스테이징 및 커밋
git add .
git commit -m "feat: 반복 표현 검출 로직 구현"

# 푸시
git push origin feature/b-rule-analysis
```

### 5-4. 커밋 메시지 규칙

팀 내 커밋 메시지를 통일하면 히스토리 파악이 쉬워진다. 아래 형식을 권장한다.

```
<타입>: <간단한 설명>

# 타입 종류
feat:     새 기능 추가
fix:      버그 수정
refactor: 코드 구조 변경 (기능 변화 없음)
test:     테스트 추가/수정
docs:     문서 변경
chore:    설정, 빌드, 패키지 관련 변경
```

예시:
```
feat: filler word 시간 구간별 밀도 분석 추가
fix: 타임스탬프 파싱 시 밀리초 누락 수정
docs: interface_contract.md에 scoring 반환 형식 추가
chore: requirements/rule_analysis.txt에 scikit-learn 추가
```

### 5-5. PR 생성

GitHub에서 PR(Pull Request)을 생성한다.

- **Base 브랜치:** `develop`
- **Compare 브랜치:** `feature/b-rule-analysis`
- **제목:** 커밋 메시지와 동일하게
- **본문:** 무엇을 변경했는지, 테스트 결과는 어떤지 간략히 작성
- **리뷰어:** 최소 1명 지정

### 5-6. 작업 종료

```bash
deactivate
```

## 6. 특수 상황별 대응

### 상황 1: 다른 팀원의 코드가 필요할 때

아직 머지되지 않은 다른 팀원의 브랜치가 필요하다면, 임시로 해당 브랜치를 가져와서 테스트할 수 있다.

```bash
# 담당자 A의 전처리 브랜치를 임시로 가져오기
git fetch origin feature/a-preprocessing
git checkout feature/a-preprocessing

# 확인 후 내 브랜치로 복귀
git checkout feature/b-rule-analysis
```

### 상황 2: develop에 다른 팀원의 코드가 머지된 후

develop이 업데이트되면 내 브랜치에 반영해야 충돌을 줄일 수 있다.

```bash
git checkout develop
git pull origin develop

git checkout feature/b-rule-analysis
git merge develop
```

충돌이 발생하면 해당 파일을 열어서 수동으로 해결한 뒤 커밋한다.

```bash
# 충돌 해결 후
git add .
git commit -m "merge: develop 반영 및 충돌 해결"
```

### 상황 3: 내 컴에서는 되는데 팀원 컴에서 안 될 때

Docker로 동일 환경을 재현해서 확인한다.

```bash
# Docker에서 문제의 코드 실행
docker compose run --rm app python -c "from konlpy.tag import Okt; print(Okt().morphs('테스트'))"
```

Docker에서 정상이면 팀원의 로컬 환경(Python 버전, Java 경로 등)에 문제가 있는 것이다. Docker에서도 실패하면 코드 자체의 문제다.

### 상황 4: 새 패키지를 설치해야 할 때

```bash
# 1. 패키지 설치
pip install 새패키지명

# 2. 자기 requirements 파일에 추가
# requirements/rule_analysis.txt 에 아래 한 줄 추가
# 새패키지명>=버전

# 3. 커밋에 포함
git add requirements/rule_analysis.txt
git commit -m "chore: rule_analysis에 새패키지명 추가"
```

## 7. 한눈에 보는 명령어 치트시트

### venv 관련

| 상황 | macOS | Windows |
|------|-------|---------|
| venv 생성 (최초 1회) | `python3 -m venv .venv` | `python -m venv .venv` |
| venv 활성화 | `source .venv/bin/activate` | `.venv\Scripts\activate.bat` |
| venv 비활성화 | `deactivate` | `deactivate` |
| 패키지 설치 | `pip install -r requirements/모듈.txt` | 동일 |
| 설치된 패키지 확인 | `pip list` | 동일 |

### Git 관련

| 상황 | 명령어 |
|------|--------|
| 최신 코드 받기 | `git checkout develop && git pull origin develop` |
| 새 브랜치 만들기 | `git checkout -b feature/이름` |
| 브랜치 이동 | `git checkout feature/이름` |
| 변경 상태 확인 | `git status` |
| 커밋 | `git add . && git commit -m "메시지"` |
| 푸시 | `git push origin feature/이름` |
| develop 변경사항 내 브랜치에 반영 | `git merge develop` |

### Docker 관련

| 상황 | 명령어 |
|------|--------|
| 전체 앱 시작 (빌드 포함) | `docker compose up --build` |
| 전체 앱 시작 (빌드 없이) | `docker compose up` |
| 전체 통합 테스트 | `docker compose run --rm app pytest tests/ -v` |
| 컨테이너 안에서 작업 | `docker compose exec app bash` |
| 종료 | `Ctrl + C` 또는 `docker compose down` |

### 테스트 관련

| 상황 | 명령어 |
|------|--------|
| 내 모듈 테스트 | `pytest tests/test_본인모듈.py -v` |
| 특정 함수만 테스트 | `pytest tests/test_본인모듈.py::함수명 -v` |
| 전체 테스트 | `pytest tests/ -v` |
| 테스트 커버리지 확인 | `pytest tests/ --cov=src -v` |
