# 의사결정 포인트

프로젝트 진행 전/중 논의하고 결정해야 할 사항들을 정리합니다.

---

## 1. 분석 항목 정의

### Q1. 어떤 분석 항목을 우선순위로 둘 것인가?

강의 품질 평가 항목이 여러 개 있는데, 어떤 것부터 구현할지:
- 불필요한 반복 표현
- 설명의 명확성
- 예시의 적절성
- 질문 유도 패턴
- 부적절한 표현

**고려사항:**
- 구현 난이도 vs 임팩트
- 데이터로 검증 가능한 항목인가?
- 클라이언트(AXP)가 가장 중요하게 생각하는 항목은?

### Q2. 각 분석 항목의 평가 기준은 어떻게 정량화할 것인가?

예: "불필요한 반복 표현"
- 몇 번 이상 반복되면 "불필요한 반복"인가?
- 문맥에 따라 다르게 판단해야 하는가?
- 스코어링 체계는 어떻게 설계할 것인가?

## 2. 데이터 처리 방식

### Q3. STT 데이터의 품질 이슈를 어떻게 처리할 것인가?

- STT 오류 (잘못 인식된 단어)
- 문장 분리 이슈
- 화자 구분 (강사 vs 수강생 질문)

**고려사항:**
- 전처리 파이프라인을 얼마나 정교하게 만들 것인가?
- 오류가 있어도 분석 결과에 큰 영향을 주지 않는가?

### Q4. 강의 메타데이터를 어떻게 활용할 것인가?

- 과목별/강사별/강의 시간별 분석이 필요한가?
- 같은 강사의 여러 강의를 비교 분석할 것인가?

## 3. 분석 정확도 개선

### Q5. LLM을 활용할 것인가? 규칙 기반으로 갈 것인가?

**LLM 활용:**
- 장: 문맥 이해, 유연한 분석
- 단: 비용, 속도, 일관성 문제

**규칙 기반:**
- 장: 빠르고 일관성 있음, 비용 절감
- 단: 문맥 이해 부족, 유연성 낮음

**하이브리드:**
- 규칙 기반으로 1차 필터링 → LLM으로 정성 분석?

### Q6. 분석 결과의 신뢰성을 어떻게 검증할 것인가?

- Ground truth 데이터가 있는가?
- 강사 피드백을 받을 수 있는가?
- 분석 결과 검증 방법은?

## 4. 리포트 생성

### Q7. 리포트 형식은 어떻게 구성할 것인가?

- 비개발자도 이해할 수 있는 형태인가?
- 시각화(그래프, 차트)를 포함할 것인가?
- 개선 제안을 어떻게 표현할 것인가?

### Q8. 대시보드 vs 리포트 파일 생성?

- 대시보드: 실시간 확인 가능, 인터랙티브
- 리포트 파일: 배포 쉬움, 아카이빙 가능

**고려사항:**
- 클라이언트(AXP)의 사용 시나리오는?
- 개발 리소스는 어느 쪽에 더 많이 필요한가?

## 5. UI/UX

### Q9. 사용자(강사)가 쉽게 이해할 수 있는 UI인가?

- 분석 결과를 직관적으로 표현하는 방법

- 평가 기준이 "클라이언트 관점의 서비스 완성도"임
- 기술적으로 흥미로운 것보다 실제 사용성이 중요
- 어디까지를 MVP로 잡을 것인가?

## 6. 팀 운영

### Q11. 역할 분담은 어떻게 할 것인가?

- PM: 일정 관리, 클라이언트 커뮤니케이션
- 개발: 분석 파이프라인, 리포트 생성
- 데이터: 전처리, 분석 항목 정의
- 발표: 발표 자료, 시연 준비

**고려사항:**
- 전원이 기술적 기여를 해야 함
- 4주라는 제한된 시간

### Q12. 중간평가(2주차)까지 무엇을 완료해야 하는가?

필수 제출물:
1. 프로젝트 기획서 (분석 항목, 데이터 처리 방식, 스코어링 체계 정의)
2. 핵심 기능 시연 영상 (3분 이내: 단일 강의 분석 결과 시연)
3. 진행 현황 보고서
